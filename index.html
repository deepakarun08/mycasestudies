The Problem

A supply chain managed
through emails and phone calls.

ZAMS — a distribution business operating across the Gulf region — was managing its entire order lifecycle manually. Customers called or emailed to place orders. Sales teams printed documents and made phone calls to operations. Logistics updates were communicated through WhatsApp messages. Every handoff was a potential failure point, and there was no single source of truth for order status across the chain.


The business needed a portal that would serve every participant — customers placing orders, sales reps reviewing them, operations teams posting to Oracle EBS, logistics tracking shipments, procurement handling returns — while keeping each role's experience focused and uncluttered. I was brought in as the sole UX designer to take this from zero to a fully specified, wireframed system ready for development.

"Our customers had no visibility into their own orders once they were placed. They'd call us, we'd call the warehouse, the warehouse would call logistics. It was a game of telephone — and things got lost."

— ZAMS Business Stakeholder, Discovery Session
📞

No Self-Service Channel

Customers had no way to place, track, or manage their orders independently. Every interaction required contacting a salesperson — creating bottlenecks, delays, and frustration on both sides.

🔗

Broken Handoff Chain

Orders moved through Customer → Sales → Operations → Logistics → Warehouse without any shared system. Each handoff relied on manual communication — email, calls, spreadsheets — and was a potential point of failure.

📦

Returns Had No Process

When goods arrived damaged, there was no structured way to report or process returns. Customers had no mechanism to upload evidence, track approval status, or follow the credit memo process.

Key Design Decisions

Five decisions that shaped
a zero-to-one system.

01

A Dashboard That Answers "What Needs Me Right Now?"

Dashboard · All Roles · Contextual Tiles

When I started designing the dashboard, the stakeholder request was straightforward: "a page with all the key metrics." But when I mapped what each role actually needed to see when they logged in, it became clear that a generic metrics dashboard would serve no one well. A customer logging in wants to know if their shipments are on track. An operations user needs to see pending approvals. An admin needs to know if any error notifications have fired.

I designed a role-aware dashboard where the tile set, notification stream, and action areas adapt to the logged-in user. Every role sees their specific KPIs — Total Orders, Pending Validations, Blocked Styles — and a "My Contact" panel showing their assigned salesperson with direct call/email actions. The "Start New Allocation" CTA is prominent for customers but completely absent for operations users who never create orders. The dashboard isn't a homepage — it's a command centre tuned to each role's primary job.

Interview talking point: "I asked every user we interviewed the same question: 'What's the first thing you check when you start your workday?' The answers were completely different for each role — and those answers became the tile architecture of the dashboard."

02

Container Auto-Suggestion — Designing for a Constraint Nobody Had Digitised

Web Order Creation · Container Logic

One of the most surprising discoveries from user interviews was that the current Excel-based order process required customers to manually verify whether their selected items would physically fit inside a shipping container. There was no tool, no calculation — just experience and guesswork. Occasionally they'd over-order, the warehouse couldn't load, and the entire shipment had to be rearranged at cost.

I designed an automatic container suggestion engine as part of the order creation flow. As the customer adds items, the system calculates total weight and volume in real time, compares against the Container Master Table settings for each supplier, and displays visual fill indicators — showing percentage of container weight and volume used. A "Suggested Container Size" badge updates dynamically. If the customer is approaching capacity, they see it before they submit, not after the warehouse calls them.

This wasn't just a UX feature — it was a supply chain intelligence layer that had never existed in the manual process, and it became one of the most valued features in stakeholder review sessions.

Interview talking point: "The most impactful UX work on this project wasn't a navigation redesign or a visual polish pass. It was turning an invisible logistics constraint — container capacity — into a real-time visual feedback system. That's where design created real business value."

03

A 9-Milestone Order Status Timeline — Not a Status Badge

Sales Order Overview · Track Shipment · Status Architecture

Early in the process, the team assumed order status could be represented with simple badges — "In Progress," "Shipped," "Delivered." But when I mapped the actual order journey with the operations team, we counted nine meaningful milestones between order creation and delivery: Web Order Created → Web Order Approved → Sales Order Generated → Order Booked → Estimated Ship Date → Estimated Arrival Date → Good Ready Date → Actual Departure → Actual Arrival.

Compressing nine real events into three generic labels would have destroyed the transparency the portal was meant to provide. I designed a horizontal milestone timeline on both the Sales Order Overview page and the Track Shipment screen — each milestone shows its date when reached, and unfilled milestones show what's upcoming. Customers can see exactly where in the journey their order sits, not just a vague status badge.

Additionally, the Track Shipment screen integrated a real-time container map via ShipsGo API — giving customers GPS-level visibility into shipments in transit.

Interview talking point: "The status timeline decision came directly from a user interview where a customer said 'I don't want to know that my order is In Progress — I want to know where in the process it actually is.' That one sentence shaped a core interaction pattern across the entire portal."

04

Role-Differentiated Document Access — Same Page, Different Permissions

Sales Invoice · Supporting Documents · Role-based Access

The Sales Invoice Detail page needed to serve two very different user types simultaneously: customers who should only be able to view and download shipping documents, and operations users who needed to both view existing documents and upload new billing documents with drag-and-drop functionality. Building two separate invoice pages would have doubled the development surface and created maintenance complexity.

I designed a single invoice page with role-aware document sections. Both roles see the "Shipping Documents" section with identical view/download capability. But the operations view adds a second "Billing Documents" section and an "Upload Documents" drag-and-drop area with file type categorisation. The page structure is identical — what changes is what's shown. This pattern repeated across multiple screens in the portal, and I documented it as a design principle in the Solution Design Document: "Same URL, role-adaptive content."

Interview talking point: "Every time I was tempted to build separate pages for different roles, I forced myself to ask: is this actually a different task, or just a different permission level on the same task? Usually it was the latter, and that distinction saved significant development complexity."

05

Notification Templates as a User-Configurable System

Manage Notifications · Admin Screen · Template Engine

The portal needed to send triggered email notifications at over a dozen workflow events — order submitted, order approved, order rejected, shipment ready, damage report received, RMA approved, and more. The standard approach would have been to hardcode these notification templates in the backend and let developers manage them. I pushed back on this for a specific reason: business workflows evolve, and hardcoded templates become bottlenecks.

I designed a fully configurable notification template management screen for admin users. Admins can create, edit, and delete notification templates by specifying Subject, Target Role, and Email Body. Crucially, the body supports dynamic variable tags — #WEB_ORDER_NUMBER#, #CUSTOMER_NAME#, #SALES_NAME#, #LINK#, #TIMESTAMP# — which the system replaces with real-time values when triggered. Admins can see a reference panel of all available tags alongside the editor.

This turned a potential developer bottleneck into a self-service admin capability — meaning the business could update notification copy, add new notification types, or adjust role targeting without a single development sprint.

Interview talking point: "Designing the notification system as admin-configurable rather than developer-hardcoded was a product thinking decision that came from understanding the client's team capacity. They had limited dev resources post-launch. Every feature I could make self-service was one fewer support ticket."

Reflection

What building
from zero taught me.

ZAMS was the most end-to-end UX engagement I've worked on — from blank-slate discovery through to a production-ready specification document. The lessons were proportionally deep.

🗺️

Process mapping is design work

The four cross-functional workflow diagrams — Customer, Sales, Operations, Returns — were the most impactful design deliverables on this project. Getting the process right before touching any screen design meant every subsequent UX decision had a solid foundation. If I'd skipped directly to wireframes, I'd have designed the wrong screens confidently.

🧩

Role complexity is best resolved early, not patched late

With 7 distinct user roles, there was constant temptation to design for the "primary" user (the customer) and adapt the rest later. Every time I followed that shortcut in early sketches, I'd hit a conflict with an internal role's needs that required rework. Designing all 7 role journeys in parallel — even roughly — was the right call.

⚙️

The container auto-suggestion was the most underestimated feature

It took me a while to convince stakeholders this was worth the complexity. They saw it as a "nice to have." After two rounds of review, when they realised it would eliminate an entire class of operational errors that had been costing them money, it became a must-have. Sometimes the designer's job is to make the business see the value in their own problems.

📄

A good SDD is a design artefact, not just a technical document

Writing the 55-page Solution Design Document forced me to think through every edge case, every validation rule, every empty state and error condition. The act of specifying it in writing surfaced about 30% of the design problems I'd missed in wireframes alone. I now treat documentation as the final phase of design thinking, not an afterthought.

Key Design Decisions

Five choices that made
the difference.

01

The 3-Step Wizard Over a Single Form

Order Type → Selection → Data Sheet

My first instinct — and the stakeholders' initial request — was a single long configuration page. It seemed efficient. But when I mapped all the inputs together, I realised the form had hard dependencies: you couldn't configure country splits before setting a product hierarchy. You couldn't reach item data without completing selection first. The sequence wasn't arbitrary — it was logical.

I introduced a 3-step wizard with a persistent progress stepper at the top. Each step is self-contained, validated before proceeding, and clearly labelled: Order Type → Selection Module → Item Data Sheet. This eliminated the "blank stare" problem where users didn't know what to fill first, and it naturally enforced data integrity — the system couldn't generate an allocation from incomplete configuration.

Interview talking point: "The wizard pattern wasn't just a UX choice — it was a data integrity decision. Enforcing sequence in the UI meant fewer validation errors in the back end. I aligned the user flow with the technical dependency graph."

02

Dependent Hierarchy Filtering with Live Context Hints

Selection Module — Country Split

The product hierarchy had four levels: Division (H1) → Department (H2) → Sub-Department (H3) → Category (H4). In Excel, analysts had to cross-reference lookup tables manually to understand what was valid at each level. In the new system, I designed cascading dropdowns where lower levels are locked until the higher level is selected.

I added a small amber contextual hint: "Lower hierarchy levels are enabled based on higher-level selection." This tiny detail — which I debated removing for being "obvious" — turned out to be one of the most praised elements in stakeholder reviews. First-time users read it. Experienced users ignored it. That's exactly what good hint text does.

Multi-select chip tags for department and category selections (BOY, GIR, BOA, BUL, BAG, HBG) also let analysts see their configuration at a glance without re-opening dropdowns — reducing cognitive load on a data-dense screen.

Interview talking point: "I spent real time on the contextual hint copy. It needed to be informative without being condescending. The principle I followed: write for the user on day three, not day one."

03

Country Split Configured on the Same Screen as Hierarchy

Selection Module — 3-Panel Layout

This was the hardest layout decision I made. Country split configuration (which countries, which split granularity, which sales factor), product hierarchy, and calculation control (buffer %, final AVG%) are three interdependent configurations. Putting them on separate steps would have created constant back-and-forth. Putting them all on one screen risked overwhelming users.

I landed on a 3-panel horizontal layout within a single wizard step: Product Hierarchy on the left, Country Split Configuration in the centre, Calculation Control on the right. This mirrored how analysts actually think — they consider hierarchy and country split simultaneously. The layout externalised that mental model into the interface.

The "Split by" toggles (Item Style, Range, Color) were a specific decision to represent a mutually-inclusive selection — not radio buttons, not checkboxes, but toggles — because analysts sometimes needed multiple split granularities enabled together.

Interview talking point: "I mapped the screens against the analysts' actual thought sequence before I drew a single frame. Layout decisions should reflect cognition, not just aesthetics."

04

A Configurable Data Table with Persistent Column Visibility

Item Data Sheet

The Item Data Sheet was where the allocation actually lived — 50 items, 25 eligible stores, 2,000 suggested allocations, and 16 blocked items surfaced in a summary bar at the top. The table itself had 25+ columns spanning fixed headers (PACK CODE, ITEM CODE, H1–H4, STORE GRID, PROMO FLAG) and optional attribute columns (ATTR1–4).

Putting 25+ columns in front of a user with no control is a recipe for overwhelm. I designed a left-panel column visibility manager with toggle-eye icons, split into "Display Headers (Fixed)" and "Optional Headers." Users could personalise what they saw without losing data — and a Save Template action preserved their configuration for future sessions.

The summary tiles (Total Items, Eligible Stores, Suggested Allocations, Blocked Items) at the top gave users instant allocation health awareness before they even looked at rows. The red "Blocked Items" tile was intentionally alarming — those needed immediate attention.

Interview talking point: "The summary tiles were designed to answer the first question every analyst asks when they land on a data sheet: 'Is this allocation healthy?' I wanted that answer visible in under a second, without reading a single row."

05

The Dashboard as a Control Tower, Not a Welcome Screen

Dashboard

The dashboard could have been a simple landing page with a "Start" button. Instead, I designed it as an operational control tower. Three status tiles at the top — Benchmarks Updated (12), Pending Validations (7, with 5 urgent), and Blocked Styles (3, +3 this week) — surface the health of the entire allocation pipeline before any action is taken.

The Recent Allocations table showed real-time allocation history by type, brand, date, and status — including Error states, which are as important to surface as Completed ones. An analyst could return to the dashboard after lunch and immediately know: has anything broken? what's pending? what needs me?

The "Start New Allocation" call-to-action was intentionally prominent but not the only thing on the screen. The dashboard had to serve the returning user, not just the first-time user.

Interview talking point: "I always design dashboards for the 90% case, not the 10% case. The 90% case here was a returning analyst, not a new user. That's why operational status came first, and onboarding CTAs came second."

Reflection

What I'd do
differently.

Building a complex enterprise tool as a solo designer is demanding and rewarding in equal measure. Looking back, here's what I learned — and what I'd change.

🔍

I'd push harder for user shadowing sessions earlier

Working from stakeholder requirements meant I was designing for a described workflow, not an observed one. A few sessions watching analysts work in Excel before any design started would have surfaced edge cases I only caught later in review.

📐

I'd design the empty states on day one

Empty states — dashboard with no recent allocations, data sheet with no filtered results — were designed last. They should have been designed alongside the filled states, because they're what users see most when they're confused.

📊

I'd push for a usability test before final handoff

We validated through stakeholder reviews, not analyst testing. Given access, I'd have run at least two task-based tests — "complete a Country Split allocation for REDTAG, BOY department" — to catch any flow friction before development locked things in.

⚡

The 3-panel layout on Selection Module was worth defending

There was pushback on putting three panels on one screen. I'm glad I held the position. The alternative — separate steps for hierarchy, country, and calculation — would have added three extra clicks per allocation run, multiplied by daily usage, multiplied by every analyst. Friction compounds.

The Problem

Customers in the dark
about their own shipments.

The business needed a self-service customer portal that could surface the right information at the right time — reducing support load, improving customer confidence, and reflecting the enterprise quality expected of an Oracle-ecosystem product.

"Our customers shouldn't need to call us just to find out if their order has shipped. We needed to give them a window into our operations — without exposing our internal complexity."

High Support Volume

🔍

Zero Fulfilment Transparency

Once an order was placed, customers had no visibility into warehouse processing, packing, or dispatch readiness. Trust eroded with every unanswered question.

📱

No Device Flexibility

The existing internal tools were desktop-only. Customers on mobile or tablet had no usable interface — a critical gap for field buyers and procurement teams on the move.

Key Design Decisions

Four choices that shaped
the portal experience.

01

Oracle Redwood as Foundation, Not Constraint

Design System Strategy

Working within Oracle Redwood's design system was a mandate, not a choice. But there's a difference between using a design system and hiding behind one. My approach was to treat Redwood as the structural skeleton — typography, colour tokens, base components — while crafting the information architecture and interaction flows entirely for this customer context.

Where Redwood had an appropriate component (data tables, status badges, form elements), I used it. Where the customer journey needed something specific — like a multi-stage order timeline or a fulfilment progress indicator — I designed custom patterns that respected Redwood's visual language without being limited by its default assumptions.

Interview talking point: "Design systems are a starting point, not a ceiling. My job was to understand Redwood deeply enough to know which rules to follow faithfully and which gaps I needed to fill with thoughtful custom patterns."

02

Order Status Architecture — Designed for Clarity, Not Completeness

Order Tracking Flow

The warehouse operations team wanted to expose every internal status: received, queued, picked, packed, quality-checked, dispatched, in-transit. That's eight states. I pushed back — not because the data wasn't real, but because customers don't need operational granularity, they need confidence.

I reduced the customer-facing status model to four meaningful milestones: Order Confirmed → Processing → Ready for Dispatch → Delivered. Behind each milestone, the system still tracked internal sub-states for operations. But the customer saw a clean, predictable journey that answered the only questions they actually cared about: "Has my order been received? Is it being processed? When does it ship?"

Interview talking point: "I ran a workshop exercise called 'What would your customer text their colleague?' Every status state we proposed had to answer that text naturally. If it couldn't, we cut it or merged it."

03

AI-Assisted Prototyping as a Stakeholder Confidence Tool

Process Innovation

This project was the first time I systematically integrated AI tools into my design workflow — not just for research synthesis, but for generating low-fidelity prototype variants quickly enough to present alternatives in the same session where requirements were discussed.

When a stakeholder raised a new requirement in a workshop, I could sketch two or three conceptual responses within minutes using AI-assisted layout tools, project them immediately, and facilitate a live discussion around real visual options rather than abstract descriptions. This compressed the feedback loop dramatically and gave stakeholders something concrete to react to — which consistently produced sharper, more actionable feedback than traditional note-taking sessions.

The AI tools didn't replace design thinking. They replaced blank-page waiting time — letting me spend creative energy on the decisions that actually required it.

Interview talking point: "AI-assisted prototyping didn't make me faster at execution — it made me faster at getting to the questions worth asking. That's a very different kind of speed."

04

Responsive-First for a B2B Audience That's Often Mobile

Device Strategy

There's a common assumption in enterprise UX that B2B users are always at a desk. The stakeholder discovery process challenged this: procurement contacts, field buyers, and warehouse liaisons frequently check order status on their phones between meetings, on the warehouse floor, or while travelling.

I designed mobile as a primary consideration, not an afterthought. Key flows — order status check, fulfilment progress, document download — were validated at 375px width before being adapted upward. The desktop experience gained density and additional data columns, but the mobile experience was complete and never felt stripped down.

Interview talking point: "For this project, I stopped asking 'how do we adapt the desktop for mobile' and started asking 'what does someone need when they have 30 seconds and a phone?' That reframe changed the priority of almost every feature on the page."

Key Design Decisions

Five decisions that defined
the platform's character.

01

Designing Four Role Experiences Without Building Four Apps

Role Architecture · Permission Design — See: Homepage (Seller) vs Admin Form screens

Sales Accelerator had four distinct user types: Sellers who consume and share content, Partners with curated access to relevant materials, Content Admins who govern the library, and Contributors who submit new content. Each had a different mental model, different primary tasks, and different risk profiles if they accessed the wrong thing.

The trap would have been to build four separate interfaces. Instead, I designed a single adaptive shell with a shared navigation structure and role-aware component states. A Seller sees a search-forward, share-optimised experience. An Admin sees the same shell — but with governance controls surfaced, content approval queues visible, and bulk management actions available. The underlying component library is the same; what changes is what's visible, what's interactive, and what's contextually relevant.

This approach also future-proofed the system — adding a fifth role would require configuration, not redesign.

Interview talking point: "I treated roles as lenses on the same data, not as separate products. The design challenge was figuring out the minimum permission delta that created meaningfully different experiences — without fragmenting the codebase or the design system."

02

Content Organisation — Taxonomy Before UI

Information Architecture — See: Homepage Navigate panel + Search Results grouping

Before touching a single screen layout, I spent significant time with the Product Manager working out the content taxonomy — how items would be categorised, tagged, and retrieved. This was the unsexy work that most designers skip, and the reason most content management UIs fail: the structure is wrong before the interface even starts.

We defined a two-level hierarchy (Collections → Items), with flexible tagging for cross-cutting themes like product line, industry vertical, and content type. The search experience was designed around this taxonomy — predictive filtering, tag-based browsing, and full-text search all relied on clean underlying structure. A beautiful search UI built on top of unstructured content is still a dead end.

Interview talking point: "IA work almost never shows up in a portfolio. But it's often the highest-leverage design decision in the project. On Sales Accelerator, getting the content taxonomy right meant every subsequent UI decision had a solid foundation to build on."

03

WCAG AA as a Design Constraint, Not a Checklist

Accessibility · WCAG 2.1 AA

Accessibility compliance on enterprise Oracle products isn't optional — it's a contractual and legal requirement. But the way teams often approach it — audit at the end, retrofit what fails — produces technically compliant, experientially broken products. I embedded accessibility thinking into the design phase itself.

Colour contrast was verified at the component level, not just page-level spot checks. Focus states were designed explicitly — visible, consistent, and aligned with Redwood's focus token — because keyboard navigation is a primary modality for many enterprise users. ARIA labels, landmark roles, and reading order were specified in design annotations handed to developers, removing ambiguity at handoff.

When I advocated for correct HTML structure with the dev team, accessibility was my strongest argument. Semantic HTML isn't just good practice — it's what makes screen readers, keyboard navigation, and assistive technologies work correctly.

Interview talking point: "WCAG compliance and good UX are not in tension — they're aligned. Every accessibility decision I made on this project also made the product clearer, more predictable, and easier to use for everyone."

04

Responsive Across 1024–1920px — No Feature Degradation

Responsive Strategy

I owned the full responsive implementation across the 1024–1920px range. This wasn't just about reflow — it was about ensuring that a sales rep on a 1024px laptop in a client lobby had the same functional capability as someone on a 1920px desktop monitor in the office. Content density changed. Feature set didn't.

Key decisions at the tighter end of the range: collapsible sidebar navigation to reclaim horizontal space, single-column content list vs. grid view toggle, and progressive disclosure of metadata on content cards. At wider breakpoints, multi-column layouts and expanded filtering panels became available. The underlying component structure accommodated both without duplication.

Interview talking point: "Responsive design in enterprise tools often gets treated as 'make it not break on small screens.' I framed it differently: every breakpoint is a context, and every context deserves a purposeful layout decision — not just a scaled-down version of the largest one."

05

Email Templates — UX-Designed and Production-Coded

Email UX · HTML Development — See: Welcome Email Template screen

I had full ownership of all Sales Accelerator email templates — from UX design through to production HTML. This is an unusual scope for a UX designer, and it shaped how I approached both disciplines. Because I was writing the HTML myself, I designed with the medium's constraints in mind from the start: no CSS grid, no flexbox, table-based layout, inline styles, limited font stacks.

Each template was designed to serve a specific trigger event — content shared with a prospect, approval request for a new submission, admin notification for library changes. The information hierarchy for each email was designed to answer one question in the subject line, one action in the preview text, and one clear CTA above the fold — regardless of email client rendering behaviour.

Testing across clients (Outlook, Gmail, Apple Mail) revealed rendering inconsistencies that I resolved at the code level — not by filing tickets and waiting, but by iterating directly. The result was a suite of templates that looked intentional and on-brand across every client without exception.

Interview talking point: "Owning email HTML development made me a better email UX designer. Knowing exactly why certain patterns break in Outlook made me design differently — simpler layouts, fewer assumptions, and better fallbacks. Constraints don't limit good design; they focus it."
